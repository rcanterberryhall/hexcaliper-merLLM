Configuration

GPU Router

merLLM round-robins all requests across both GPUs. Both GPUs normally load the same DEFAULT_MODEL. When a request specifies a different model, merLLM waits for an idle GPU, swaps it to the requested model, and routes subsequent matching requests there.

Reclaim — when a GPU running a non-default model has been idle for RECLAIM_TIMEOUT seconds (default 300), merLLM reloads the default model and resumes round-robin on both GPUs.

Health tracking — if a GPU fails, it is marked degraded and excluded from routing. Health probes use exponential backoff (HEALTH_BACKOFF_BASE → doubled each attempt → capped at HEALTH_BACKOFF_CAP). After HEALTH_FAULT_TIMEOUT seconds of continuous failure, the GPU is declared faulted. Recovery requires a manual reset via the Overview tab.

Default model change — changing the default model in Settings requires confirmation. merLLM waits for each GPU to go idle before reloading.

GPU Queue

Every GPU request is tracked in a single unified queue split across five priority buckets that drain strictly top-down. A lower bucket only dispatches when every bucket above it is empty.

1. chatReal-time chat tokens to the user. Only this.
2. reservedEmpty. Reserved for future use.
3. shortLive scan analyze, situation synthesis during a scan, contacts parsing, on-demand "regenerate" clicks.
4. feedbackLLM work spawned because a background job produced something that needs timely downstream processing.
5. backgroundBulk: reanalyze items, end-of-run briefings, lancellmot extractor, document upload/indexing.

Callers declare their bucket with X-Priority: <name>. chat requests proceed immediately if a GPU is available; if both GPUs are busy, they wait up to INTERACTIVE_QUEUE_TIMEOUT seconds (default 30) before returning 503. All other buckets wait indefinitely. The legacy interactive and batch values are still accepted as aliases for chat and background during the client migration.

Client apps identify themselves with the X-Source header (e.g. lancellmot, parsival). Requests without this header appear as direct.

Call-site rule — when a new LLM call site is added in any repo, read this table, pick the bucket, and cite it in a code comment. Keeping the bucket assignment explicit at every call site is what prevents the tiers from drifting back into an accidental mix.

Transparent wait — when a request is queued, a queue_status event is emitted before generation starts with the reason and an estimated_wait_seconds hint. LanceLLMot and Parsival display this as a waiting indicator. If the wait exceeds QUEUE_HEARTBEAT_INTERVAL_SECONDS (default 20), the streaming proxy then emits periodic {"type":"queue_status","waiting":true,"elapsed_seconds":N} keepalive lines so a caller using requests.post(timeout=60, stream=True) does not trip its read-gap timer during a long background-bucket drain.

Batch Jobs

Batch jobs run at bucket 5 (background) — they only dispatch when every higher bucket is empty. Chat, short foreground work, and feedback work all take priority.

Automatic retry — failed jobs are retried up to BATCH_MAX_RETRIES times (default 2, giving 3 total attempts) with exponential backoff (30s, 120s). On final failure, the error messages are stored on the job record.

Notifications — when a batch job completes, merLLM sends a webhook POST to NOTIFICATION_WEBHOOK_URL (if configured) with job ID, source app, status, timestamps, and prompt preview. Works with Slack webhooks, ntfy.sh, Gotify, or any JSON POST endpoint. The dashboard SSE stream also receives completion events in real time.

Submit a job from the Batch Jobs tab or via the API: 

POST /api/batch/submit
{
  "source_app": "parsival",
  "prompt": "Analyze...",
  "model": "qwen3:32b"   // optional
}

GET  /api/batch/status/<id>
GET  /api/batch/results/<id>
POST /api/batch/<id>/cancel
POST /api/batch/<id>/requeue

Prompts exceeding BATCH_MAX_PROMPT_LEN (default 100,000 chars) are rejected with 422. Jobs run as soon as a GPU slot is available (at low priority). Failed jobs can be requeued; cancelled jobs can only be resubmitted.

My Day Panel

The Overview tab includes a "My Day" panel that aggregates attention items from across the ecosystem:

From Parsival — new/investigating situations, overdue follow-ups, unread high-attention items.
From LanceLLMot — pending deep analysis results, escalation results awaiting review.
From merLLM — completed batch jobs not yet retrieved, active warnings (GPU temp, Ollama down, transition failures).

Each card links to the relevant item in the relevant app. If an app is unreachable, that card shows "offline" instead of failing the whole panel.

Configure companion service URLs via PARSIVAL_URL and LANCELLMOT_URL in Settings.

Ollama Proxy

merLLM is a drop-in OLLAMA_BASE_URL replacement. Point LanceLLMot and Parsival at http://<host>:11400 and they require no other changes.

Supported endpoints:

POST /api/generateRound-robin, routed by model
POST /api/chatSame routing
POST /api/embeddingsRound-robin
GET /api/tagsModel list (GPU 0)
POST /api/showGPU 0
POST /api/pullGPU 0 (shared store)
GET /api/psAggregated from both

Set X-Priority: background (or chat / short / feedback) on any request to place it in the corresponding bucket without submitting a formal batch job. See the GPU Queue card above for the call-site rule and bucket definitions.

Reasoning-model num_ctx pin — /api/chat and /api/generate auto-fill options.num_ctx = 8192 for any qwen3:* request whose body omits it (callers can override by passing an explicit value). Without this pin, Ollama auto-fits the context window to the model's training maximum (32 k for qwen3:32b) on a freshly empty 24 GB Tesla P40, forces CPU offload of the output layer, and turns ~330 ms reloads into ~12 s reloads. The dispatcher's swap warm-load applies the same pin so both code paths converge on the same KV-cache size.

Settings

All settings can be changed live from the Settings tab — no restart required. Changes are persisted in SQLite and applied immediately to the running process.

Key settings:

ollama_0_url / ollama_1_urlOllama instance URLs
default_modelDefault model for both GPUs (change requires confirmation)
reclaim_timeoutIdle seconds before reloading default model (default 300)
health_backoff_baseInitial health probe interval (default 10s)
health_backoff_capMax health probe interval (default 300s)
health_fault_timeoutSeconds before declaring GPU faulted (default 1800)

SSH Terminal

The SSH tab provides a full browser terminal via xterm.js. To enable it:

1. Set SSH_USER in docker-compose.yml (or via Settings).
2. Place your private key at ./data/ssh_key on the host (mounted as /data/ssh_key inside the container).
3. The key must be accepted by the target host's authorized_keys.

The connection is authenticated by SSH key — no passwords are stored or transmitted. Access is gated by Cloudflare Access at the nginx layer when deployed behind a tunnel.

VNC Viewer

The VNC tab proxies a WebSocket connection to the configured VNC server. To use it:

1. Start a VNC server on the host (e.g. TigerVNC or x11vnc).
2. Set VNC_HOST and VNC_PORT in settings if different from defaults (host.docker.internal:5900).
3. noVNC's rfb.js must be available — either served from the host or loaded via CDN. Without it the viewer shows a placeholder message.

merLLM acts only as a WebSocket-to-TCP proxy; it does not start or manage the VNC server.

Metrics Charts

The Metrics tab renders time-series line charts (via Chart.js) for CPU utilization, RAM usage, GPU utilization (per card), GPU VRAM (per card), GPU temperature (per card), disk I/O, and network throughput.

Data is polled from /api/merllm/metrics/history on tab activation and on range change. Default view: last 1 hour. Range selector buttons: 1h / 6h / 24h / 7d.
